## Technical Analysis

### ACP Implementation Patterns
For developers building ACP-compatible backends:

**RESTful Approach (Recommended for most):**
```
POST /acp/v1/checkout
Headers: X-Agent-Identity, X-ACP-Version
Body: { product_id, quantity, shipping_address, payment_method }
Response: { checkout_url, session_id, expires_at }
```

**MCP Server Approach (For agent-native storefronts):**
- Expose your catalog as MCP resources
- Implement `checkout` as an MCP tool
- Handle payment delegation to Stripe via ACP
- Reference: [Stripe ACP + MCP Guide](https://docs.stripe.com/agentic-commerce/protocol)

### Agent Gateway Pattern Emerges
Solo.io and other API gateway vendors are shipping "Agent Gateway" configurations:
- **Rate Limiting:** Per-agent-identity, not per-IP
- **Authorization:** Integration with Warden keychains for wallet-based auth
- **Observability:** MCP-aware tracing (tool calls, context window usage)
- **Security:** Prompt injection detection at the gateway layer

## Security Advisory

### MCP Tool Shadowing Vulnerability
* **CVE:** Pending assignment
* **Issue:** Malicious MCP servers can register tools with names similar to trusted tools (e.g., `github_commit` vs `github-commit`)
* **Impact:** Agent may invoke attacker-controlled tool believing it's legitimate
* **Mitigation:** Pin tool names explicitly in agent configurations. Use AAIF-verified server registry.
* **Reference:** [Cerbos Security Analysis](https://www.cerbos.dev/news/securing-ai-agents-model-context-protocol)
